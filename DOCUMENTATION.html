Architecture générale

1.1 Vue d'ensemble

Le site CRTI-B est composé de deux applications distinctes qui communiquent via l'API REST de PayloadCMS.

🖥 Backend CMS

PayloadCMS 3.76
Next.js 15 · Port 3000
Interface d'admin
API REST + GraphQL

⇄

🌐 Frontend

Next.js 16.1
Port 3001
Tailwind CSS v4
SSR + ISR

⇄

🗄 Base de données

PostgreSQL 16
Port 5432
DB: certibbackend
User: postgres

ComposantTechnologiePortRôle
CMS BackendPayloadCMS 3.76 + Next.js 153000Administration du contenu, API REST
FrontendNext.js 16.1 + Tailwind CSS v43001Site public visible par les visiteurs
Base de donnéesPostgreSQL 16.145432Stockage de toutes les données
Gestionnaire de paquetspnpm—Installation des dépendances

1.2 Flux de données

1
Admin crée du contenu — L'administrateur se connecte à localhost:3000/admin et crée/modifie des pages, actualités, médias, etc.
↓
2
PayloadCMS sauvegarde — Les données sont stockées dans PostgreSQL. Un hook revalidatePage déclenche la revalidation du cache frontend.
↓
3
Frontend interroge l'API — localhost:3001 appelle /api/cms/[collection] (proxy) qui redirige vers localhost:3000/api/[collection].
↓
4
Rendu côté serveur — Next.js génère le HTML côté serveur (SSR) avec les données fraîches et l'envoie au visiteur.
↓
5
Visiteur reçoit la page — La page HTML statique/SSR est rendue dans le navigateur avec le bon contenu.

2.2 Collections (base de données)

12 collections définissent la structure complète des données :

📄 Pages

Pages du site avec éditeur de blocs (20+ types). Hiérarchie parent/enfant, ordre de menu, icône, SEO.

📰 News (Actualités)

Articles avec titre, slug, contenu richText, date de publication, rubrique (5 catégories), image.

🖼 Media

Images, PDFs, vidéos uploadés. Stockés dans /public/media/. Accès via /api/media/.

🤝 Partners

Partenaires et associations avec logo, lien externe, type (partner/association), ordre d'affichage.

🎓 Formations

Catalogue de formations BIM. Masquée du sidebar (hidden:true) — modifiable en DB ou via API.

📹 Videos

Vidéos YouTube ou fichiers internes. Type (youtube/internal), URL source, ordre d'affichage.

📊 ActivityReports

Rapports d'activité annuels. Champs : année, fichier PDF.

⏳ TimelineItems

Éléments de la ligne du temps (Notre histoire). Année, titre, ordre.

📋 Clauses

Clauses techniques et contractuelles (CTG). Numéro, titre, date, version, PDF/RTF/DOC.

📧 NewsletterSubscribers

Abonnés à la newsletter. Email, statut (active/inactive), token de désabonnement.

👤 Users

Comptes administrateurs. Email + mot de passe. Authentification JWT via PayloadCMS.

2.5 Éditeur de pages – Système de blocs

Chaque page dispose d'un champ Layout (constructeur de pages) permettant d'ajouter, réordonner et configurer des sections.

Le champ Content est un éditeur richText Lexical enrichi (texte + images + callouts + accordéons inline).

Ordre du menu : Les pages avec menuOrder < 900 et sans page parente apparaissent dans la navigation principale. Les pages enfant s'affichent dans les menus déroulants.

3.2 Layout global & navigation

Fichier : src/components/layout/GlobalLayout.tsx

1
Récupère toutes les pages publiées depuis l'API CMS (depth=1, limit=1000)
2
Construit la hiérarchie parent → enfant → petit-enfant
3
Filtre : n'affiche que les pages avec menuOrder < 900 dans la nav
4
Rend : <HeaderWithNav /> + <main>{children}</main> + <Footer />

HeaderWithNav

  • Navigation desktop : barre horizontale avec dropdowns à 3 niveaux
  • Navigation mobile : menu tiroir (hamburger) avec accordéons
  • Icônes SVG inline pour chaque item de menu (12 types d'icônes)
  • Bouton de recherche → ouvre SearchModal
  • Logo CRTI-B cliquable → retour à l'accueil
  • Header sticky (position: sticky; top: 0; z-index: 50)

Footer

  • 3 colonnes : Logo + description, Coordonnées, Liens utiles
  • Fond sombre #2F3E46, icônes teal #08AA86
  • Barre de copyright avec Mentions légales + Contact

3.3 Système de pages dynamiques

Toutes les URLs non définies explicitement sont gérées par app/[...slug]/page.tsx :

1
L'URL /mpu/formation-mpu extrait le slug mpu/formation-mpu
2
Appel à getPageBySlug("mpu/formation-mpu") via l'API CMS
3
La page CMS retourne son title, layout[] (blocs) et content (richText)
4
PageLayout rend : Breadcrumb + Hero titre + BlockRenderer + RichText content
Important – Slugs : Les slugs ne doivent pas contenir d'espaces (ex: formation mpu → 404). Utiliser uniquement des tirets : formation-mpu. Le hook beforeValidate dans Pages.ts génère automatiquement les slugs depuis le titre.

3.4 API Proxy

Le frontend ne contacte jamais directement le CMS depuis le navigateur. Toutes les requêtes passent par le proxy Next.js :

// app/api/cms/[...path]/route.ts
GET  /api/cms/pages?where[slug][equals]=contact
  → http://localhost:3000/api/pages?where[slug][equals]=contact

GET  /api/media/file.jpg
  → http://localhost:3000/media/file.jpg

La fonction getMediaUrl(media) retourne /api/media/[filename] côté client ou http://localhost:3000/media/[filename] côté serveur.

Section 04

Système de blocs (Page Builder)

4.1 Architecture BlockRenderer

Fichier : src/components/home/BlockRenderer.tsx

Ce composant reçoit un tableau blocks[] et rend le composant correspondant à chaque blockType.

Principe : Dans le CMS, l'admin ajoute des blocs à une page. Chaque bloc a un blockType (slug) et des champs configurables. Le frontend lit ces blocs et affiche le composant React correspondant.

Section 07

Fonctionnalités spéciales

7.1 Newsletter

1
Visiteur saisit son email dans le NewsletterBlock
2
POST /api/newsletter → sauvegarde dans la collection newsletter-subscribers avec un token unique
3
Admin publie un article News → bouton "Envoyer la newsletter" dans le CMS
4
PayloadCMS envoie l'email via Nodemailer à tous les abonnés actifs avec lien de désabonnement
5
Clic sur désabonnement → /api/newsletter/unsubscribe?token=xxx → statut passe à inactive

7.2 Formulaire de contact

1
Visiteur remplit le formulaire (nom, email, sujet, message) sur /contact
2
POST /api/contact (route API Next.js côté serveur)
3
Email envoyé via Nodemailer à contact@crtib.lu avec les données du formulaire
4
Affichage du message de confirmation → "Message envoyé !"

La page contact est maintenant une page CMS standard avec un contactBlock, comme toutes les autres pages.

7.3 Recherche

  • Bouton loupe dans le header → ouvre SearchModal
  • Appel à /api/search?q=terme
  • Recherche dans les titres et slugs de pages via l'API CMS
  • Résultats cliquables avec navigation directe

7.4 Actualités & rubriques

La collection News dispose de 5 rubriques :

7.6 Revalidation du cache

Lorsqu'une page est modifiée dans le CMS, le hook revalidatePage (dans crtib-cms-main/src/hooks/revalidateFrontend.ts) appelle POST /api/revalidate sur le frontend avec le slug de la page modifiée. Next.js invalide alors le cache ISR de cette page pour que les visiteurs voient le nouveau contenu immédiatement.

Section 08

Guide d'administration

8.1 Créer ou modifier une page

1
Aller dans Pages dans la sidebar admin → cliquer "Créer nouvelle Page"
2
Remplir Titre (requis) → le slug se génère automatiquement. Vérifier le slug (pas d'espaces, pas d'accents)
3
Choisir la Page Parente (si sous-menu) et l'Ordre du Menu (1-899 visible, 900+ caché)
4
Dans Sections de la page : ajouter des blocs (cliquer "+", choisir le type, configurer)
5
Cliquer Publier (ou Sauvegarder brouillon) → la page est visible sur le frontend immédiatement
Slugs : Ne jamais mettre d'espaces dans un slug. Utiliser uniquement des lettres minuscules, chiffres et tirets. Exemple : performance-energetique et non performance énergétique.

8.2 Publier une actualité

1
Aller dans Actualités → "Créer nouvelle Actualité"
2
Remplir titre, date de publication, choisir la rubrique
3
Ajouter une image de couverture (optionnel mais recommandé)
4
Rédiger le contenu dans l'éditeur richText (supports images, titres, listes, liens)
5
Cliquer Publier. Optionnel : envoyer la newsletter aux abonnés depuis cet article

8.3 Ajouter un bloc à une page existante

  1. Ouvrir la page dans le CMS admin
  2. Défiler jusqu'à Sections de la page
  3. Cliquer le bouton + Ajouter un bloc
  4. Choisir le type de bloc dans la liste (20 types disponibles)
  5. Remplir les champs du bloc (titre, contenu, images…)
  6. Réordonner les blocs par drag-and-drop si nécessaire
  7. Sauvegarder / Publier
